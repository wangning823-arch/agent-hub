# Phase 0 Design — 初步模块化与改造蓝图

目标
- 为后续阶段的模块化改造建立清晰的职责界限与接口约定，确保第一阶段变更对现有行为无风险地推进。
- 产出可直接用于团队对齐、评估与后续实现任务分解。

总体原则
- 最小化风险：不破坏现有 HTTP API 行为，逐步拆分职责，避免一次性大改。
- 明确边界：将路由、WebSocket、会话、持久化、权限等职责分层，方便后续独立测试。
- 可扩展：优先设计可替换的持久化与存储接口，未来支持数据库或分布式存储。

阶段性边界
- 阶段0（快速 wins）: 搭建改造骨架、明确接口、创建文档与 scaffolding，不变更现有运行行为的前提下为后续实现打好结构基础。
- 阶段1: 后端模块化与解耦实现（路由、控制器、服务、WebSocket hub 分离）
- 阶段2: 数据持久化演进与 DAO

阶段0 产出物
- backend 目录下的模块化骨架设计草案（路由/控制器/服务/中间件/WS hub 的初步分层结构）
- 相关的 README/文档补充，明确接口约定与文件结构
- frontend 的初步前端分层计划草案（自定义 Hook 的接口草案）
- 供后续实现的 skeleton 代码（未修改现有行为的最小实现）

验收标准
- 设计文档与骨架代码存在且可被团队成员理解，且未来实现可直接对接现有服务器/前端代码入口。
- 不改变现有 API 路径与行为，确保阶段0上线后系统仍能正常运行（在现有场景下应无新错）。
